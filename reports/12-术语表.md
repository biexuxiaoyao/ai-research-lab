## 术语表 (Glossary)

> 本术语表收录报告中出现的核心概念、前沿术语、缩略语及其定义。按拼音排序，每个术语标注首次出现的章节及英文对应词。

---

### A

**Agent（智能体）**
：具备自主决策和行动能力的 AI 系统。在软件工程语境中，指能够自主编写代码、执行命令、调用工具并完成多步骤任务的 AI 程序。详见第十四章。

**Agent Engineering（Agent 工程）**
：专注于设计、编排、治理和运维 AI Agent 系统的新兴工程学科。预计 2026-2028 年从传统软件工程中分化出独立职业路径。详见第九章。

**Agentic CI/CD**
：由 AI Agent 自主驱动的持续集成和交付流水线。Agent 自动执行代码生成、测试、审查和部署决策，人类角色从操作者转为审批者。详见第七章。

**Agentic Debt**
：因 AI Agent 自主决策和代码生成而积累的技术债务。与传统技术债务不同，Agentic Debt 的来源不是人类的妥协决策，而是 Agent 缺乏全局上下文导致的次优选择积累。详见第十二章。

**AGENTS.md**
：仓库级 Agent 指令文件，定义 Agent 在该仓库中的行为规范、工程约定和项目上下文。与 CLAUDE.md 等同类文件共同构成指令文件生态。详见第一章。

**AI Coding Three Eras（AI 编码三个时代）**
：Cursor 提出的 AI 编码演化分期——第一代 Tab 自动补全（2022-2024）、第二代同步 Agent（2024-2025）、第三代云端 Agent（2025-2026）。三个时代对应三种人机关系：助手→协作者→编排者。详见引言。

**API Contract as Source of Truth（API 契约即真相源）**
：在 AI 辅助开发中，API 规范（OpenAPI/GraphQL Schema）而非代码实现成为系统行为的权威定义。AI Agent 生成代码时以 API 契约为约束边界。详见第四章。

**Architecture Drift（架构漂移）**
：多个 AI Agent 并行开发时，由于缺乏全局架构感知，各自引入不一致的模式和技术选型，导致系统架构逐渐偏离原始设计。详见第四章。

**Automation Paradox（自动化悖论）**
：AI 运维越强大，系统可能越脆弱的反直觉现象——自动化减少了人为干预机会，但增加了故障的复杂性和影响面。详见第八章。

### B

**Bottleneck Displacement（瓶颈全局位移）**
：当 AI 使编码不再成为瓶颈时，软件工程的约束从前端编码位移至需求定义、规格验证和治理决策。这是本报告的核心论断之一。详见第十二章。

### C

**Circular Verification（环形验证）**
：当同一 AI 模型既生成代码又生成测试时，错误可能被相互强化而非抵消——代码的错误假设被复制到测试中，导致虚假通过。破解方法包括异质模型审查和突变测试。详见第六章。

**CLAUDE.md**
：项目级 Agent 指令文件，定义项目结构、技术栈、开发约定和 Agent 行为规范。由 `/init` 命令生成，是 Agent 理解项目上下文的关键机制。详见第一章及第十四章。

**Code Coverage Signal Devaluation（覆盖率信号贬值）**
：AI 可以轻易生成大量高覆盖率的测试代码，但覆盖率数字的提高不再与代码质量正相关。传统的"覆盖率=质量保障"信号在 AI 时代失效。详见第六章。

**Context Window Engineering（上下文窗口工程）**
：系统性地设计和管理 LLM 上下文窗口内容的工程实践，包括上下文压缩、信息密度优化、分层上下文注入等技术。详见第十八章。

### D

**Design Token（设计令牌）**
：一种标准化、跨平台的设计决策表达方式。以键值对形式存储颜色、字体、间距等设计属性，使设计系统可以跨技术栈一致应用，尤其适合 AI 辅助生成前端代码。详见第二章。

**DESIGN.md**
：意图工程的核心载体——一种声明式的设计意图描述文件，定义产品应该"是什么"和"为什么"，而非"怎么做"。与 AGENTS.md（行为规范）互补，共同构成 AI Agent 的输入上下文。详见第十八章。

### H

**Hallucination Citation（幻觉引用）**
：AI 生成看似真实但实际不存在的参考文献。通常具有真实作者名+虚构标题、真实期刊名+不存在的卷期页码等特征。详见第三章"推荐引用工具指南"。

**Harness（Agent 运行时框架）**
：为 AI Agent 提供工具注册、权限控制、状态管理、上下文维护和输出验证的基础设施层。是 Agent 与外部世界交互的"操作系统"。详见第十四章。

**Heterogeneous Model Review（异质模型审查）**
：使用不同于代码生成模型的独立 AI 模型来审查代码质量或测试有效性，以打破同一模型的"回音室效应"。详见第六章。

### I

**Intent Engineering（意图工程）**
：比提示工程更高层次的学科——关注如何精确、完整、可验证地表达人类的意图，使 AI Agent 能够准确地理解和执行。DESIGN.md 是其核心实践。详见第十八章。

### L

**LLM-as-Judge**
：使用一个独立的 LLM 作为评估器，对 AI 生成内容的质量、安全性、相关性等进行自动化打分和评判。详见第十七章。

### M

**MCP（Model Context Protocol，模型上下文协议）**
：Anthropic 发布的开源协议，标准化 AI 模型与外部工具、数据源之间的交互方式。类似 AI Agent 的"USB 接口"，使不同 Agent 和工具能够互操作。详见第十四章。

**Meta-Harness（元治理层）**
：在 Agent-Harness 之上建立的自动化治理层，负责监控 Agent 行为、检测偏差、执行合规策略和自动化干预。详见第九章。

**Mutation Testing（突变测试）**
：通过向代码中注入人为错误（突变体），验证测试套件是否能够检测到这些错误的测试方法。在 AI 时代成为比代码覆盖率更可靠的测试质量指标。详见第六章。

### N

**Normalization of Deviance（规范化越轨）**
：Simon Willison 提出的概念——开发者逐渐接受"不再逐行审查 AI 生成的代码"作为新常态，即使知道这违反传统工程规范。是 AI 时代安全风险的重要来源。详见第九章。

### P

**Prompt Injection（提示注入）**
：攻击者通过在输入中嵌入恶意指令，诱导 AI Agent 执行非预期操作的安全攻击方式。是 AI 应用面临的最普遍且最难防御的安全威胁之一。详见第十章。

### S

**SDD（Spec-Driven Development，规范驱动开发）**
：以形式化或半形式化的规范文档驱动软件开发的新型方法。开发流程为：Constitution → Specify → Plan → Tasks → Implement。代表工具包括 GitHub Spec-kit。详见第一章。

**Signal System Collapse（信号体系崩溃）**
：传统软件工程质量信号（代码覆盖率、提交频率、审查时间等）在 AI 辅助开发环境下系统性失效，需要建立新的质量度量体系。详见第十二章。

**Slopsquatting（Slop 劫持）**
：一种新型 AI 供应链攻击方式——攻击者利用 AI 编码工具的上下文窗口，诱导 Agent 引入恶意依赖或配置。名称源自 typosquatting（拼写劫持）。详见第十章。

### T

**Text-to-SQL**
：将自然语言查询转换为 SQL 语句的 AI 技术。基准测试与实际生产环境之间存在显著的准确率鸿沟（benchmark 90%+ vs. production ~60-70%）。详见第五章。

### V

**Vibe Coding（氛围编码）**
：Andrej Karpathy 提出的概念——开发者通过自然语言描述意图（"vibe"），让 AI 完成全部代码实现，开发者仅在最终产物不符合预期时进行干预。是 AI 编码从辅助到自主的中间态。详见第九章。

**Vibe Coding → Agentic Engineering 演化谱系**
：从完全人类编码到完全自主 Agent 工程的能力连续体，包含六个等级：Manual → Copilot → Vibe Coding → Spec-Guided → Agentic Engineering → Autonomous。详见第九章。

### E

**EU AI Act（欧盟人工智能法案）**
：Regulation 2024/1689，全球首部全面 AI 监管法律，2024 年 8 月生效，分阶段执行。对高风险 AI 系统施加全生命周期合规义务，最高罚款 €3500 万或全球年营业额的 7%。详见第十一章。

**Embedding（嵌入向量）**
：将文本/代码转换为高维数值向量的技术，是 AI 语义搜索、代码相似度检测和 RAG 系统的基础。详见第五章（数据层）与第十七章（可观测性）。

### F

**FDE（Forward Deployed Engineer，前向部署工程师）**
：2026 年最难招聘的混合角色之一——在客户环境中配置产品、训练 Agent、捕获边界条件、将部署作为一等工程问题处理。详见第九章。

**Fleet Supervisor / Fleet Fixer**
：管理 50+ AI Agent 实时状态的"空中交通管制员"（Supervisor），或调试"机器之间对话"理解多 Agent 涌现行为的角色（Fixer）。2026 年新兴角色。详见第九章。

**FinOps for AI（AI 成本管理）**
：将云计算 FinOps 理念扩展到 AI Token 消耗管理，按 Feature/Team/Agent 维度进行成本归因和优化。详见第十七章。

**Formal Verification（形式化验证）**
：使用数学方法（TLA+、Lean 4、Coq）证明系统正确性的技术。AI 降低了形式化验证的学习成本，使其从学术进入工业（安全关键系统）。详见第六章。

### G

**GPAI（General-Purpose AI，通用人工智能模型）**
：EU AI Act 对 GPT、Claude 等通用 AI 模型的法律分类，需承担技术文档、版权合规和训练数据摘要等义务。详见第十一章。

**GPL / AGPL（GNU 通用公共许可证）**
：具有"传染性"的开源许可证——基于 GPL 代码的衍生作品必须以相同许可证发布。AI 训练中摄入 GPL 代码后可能在输出中复现 GPL 片段，造成"许可证污染"。详见第十一章。

**Guardrails（守卫/护栏）**
：对 AI Agent 行为的结构性约束机制——不同于 Prompt 中的"建议"，Guardrails 在工具层、权限层或策略引擎层强制执行。详见第十章、第十四章。

### H

**Human-in-the-Loop / Human-off-the-Loop（人在回路中 / 人退出回路）**
：两种人机关系模式。HITL = 人类保持在决策环中，审批关键操作。HOTL = AI 自主执行，人类仅在事后审计。EU AI Act 要求高风险系统保持 HITL。详见第十一章。

### K

**K 型分化**
：AI 时代设计品味的两极分化现象——基础设计被 AI 和开源组件库压缩至底线，高端设计溢价暴涨。详见第二章。

### O

**OWASP（开放式 Web 应用安全项目）**
：发布 AI 安全领域最权威的风险分类——OWASP Top 10 for LLM Applications（2025）和 OWASP Top 10 for Agentic AI（2025）。详见第十章。

**OpenTelemetry（OTel，开放遥测标准）**
：CNCF 的分布式追踪标准，已被 GenAI 社区采纳为 Agent Trace 的语义约定基础。OpenInference（Arize）定义了 GenAI Span 标准化属性。详见第十七章。

**Observability（可观测性）**
：区别于传统监控——三支柱 Trace（链路追踪）+ Log（结构化日志）+ Metrics（量化指标）。AI Agent 场景还需增加推理链追踪、LLM-as-Judge 和成本归因。详见第十七章。

### P

**PR（Pull Request，拉取请求）**
：代码贡献的标准协作机制。AI 时代 PR 量 5× 增长导致审查时间攀升 400%+，PR 的本质从"编码审查"变为"AI 输出质量验证"。详见第七章。

**Platform Engineering（平台工程）**
：构建内部开发者平台（IDP）以降低认知摩擦、标准化工具链的工程学科。当 AI 编码成本趋近零时，平台工程的 ROI 显著上升。详见第七章、第十二章。

**Prompt Engineering（提示工程）**
：设计和优化 LLM 输入格式以获得期望输出的工程实践。2026 年正从"写更好的 Prompt"进化为**上下文工程**和**意图工程**——系统性地管理上下文窗口和表达人类意图。详见第十八章。

**Property-Based Testing（属性测试）**
：不测试具体输入输出，而测试数学性质（如"排序后数组长度不变"）的测试方法。在 AI 生成的测试套件面前比单测更难被"欺骗"。详见第六章。

### Q

**QA（Quality Assurance，质量保障）**
：AI 时代的 QA 角色从"执行测试"转型为"质量策略师 + AI 测试裁决者"，关注测试 ROI 评估、突变测试设计和异质模型审查。详见第六章。

### R

**RAG（Retrieval-Augmented Generation，检索增强生成）**
：在 LLM 推理前从外部知识库检索相关文档注入上下文的架构。生产环境三大失败模式（2026 年 arXiv:2605.03275）：上下文污染、Schema 误解、查询歧义。详见第五章。

**ROI（Return on Investment，投资回报率）**
：AI 编码工具的经济效益度量。Figma Dev Mode 的三年 ROI 报告为 351%，但 AI ROI 的关键变量是审查成本和修复成本而非编码速度。详见第二章。

### S（续）

**SAST / SCA（静态应用安全测试 / 软件组成分析）**
：SAST 检测源代码中的安全漏洞；SCA 识别第三方依赖的已知漏洞和许可证。在 AI 代码流水线中，两者正从"可选"变为 CI 标配。详见第十章。

**SBOM（Software Bill of Materials，软件物料清单）**
：列出软件中所有组件的标准化清单。AI-BOM 是其 AI 扩展——记录 AI 生成的代码段及其来源（模型版本、Prompt、生成时间）。详见第十章。

**SDLC（Software Development Life Cycle，软件开发生命周期）**
：软件从需求到退役的全过程。本报告的核心论点是：AI 迫使整个 SDLC 从"以编码为中心"重构为"以规格、验证和治理为中心"。详见全文。

**SRE（Site Reliability Engineering，站点可靠性工程）**
：2026 年 SRE 正从"第一响应者"进化为"系统编排者与 AI 治理者"。Gartner 预测 2029 年 85% 企业使用 AI SRE 工具。详见第八章。

**SLSA（Supply-chain Levels for Software Artifacts，软件制品供应链级别）**
：Google 提出的软件供应链安全框架（L0-L3）。AI 代码来源的 SLSA 级别检测正在成为新需求。详见第七章、第十章。

### T（续）

**Token（令牌 / 词元）**
：LLM 计费和上下文窗口的基本单位。不同模型对相同文本的 Token 化效率差异可达 35%。Token 成本是 AI 软件工程的 FinOps 核心指标。详见第十五章、第十七章。

**TCO（Total Cost of Ownership，总拥有成本）**
：AI 生成代码的完整经济模型——Token 成本 + Review 成本 + 修复成本 + 维护成本的总和。纯粹的"编码速度"指标严重低估了 AI 代码的真实成本。详见各章经济效益章节。

### W

**Wiki-Link（`[[双向链接]]`）**
：Markdown 中的双向链接语法（`[[页面名]]`），是 Markdown 工程化从"文件夹层级"到"网络化知识管理"的范式突破。需要在工具链兼容性和知识网络价值之间做权衡。详见第十三章。
