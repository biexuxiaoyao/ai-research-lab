## 实践检查清单与最佳实践（附录）

> 本附录将报告各章的关键行动建议提炼为可操作的检查清单，方便读者在工程实践中对照执行。

---

### P.1 需求工程：实施 SDD 的检查清单

**启动阶段**：

- [ ] 选定一个试点项目（建议：中复杂度、1-3 人周的编码量）
- [ ] 在仓库根目录创建 `AGENTS.md` 或 `CLAUDE.md`
- [ ] 编写 `Constitution` 文档（项目原则与不可妥协的约束）
- [ ] 选择 SDD 工具链（推荐：GitHub Spec-kit 作为起点）
- [ ] 为第一个 Feature 编写 `Spec` 文档（含预期行为、边界条件、验收标准）

**执行阶段**：

- [ ] 在 Agent 执行前先让人 Review Spec
- [ ] 对 Agent 生成的 `Plan` 做完整性检查（是否遗漏边界情况？）
- [ ] 将 `Tasks` 拆分粒度控制在每个 Task 不超过 30 分钟 Agent 执行时间
- [ ] 建立 "Spec→Plan→Tasks→Code→Test" 的追溯链
- [ ] 每个 Sprint 回顾 Spec 与实际实现的偏差

**规模化阶段**：

- [ ] 建立 Spec 模板库（常见 Feature 类型的 Spec 模板）
- [ ] Spec 版本化（与代码同仓库，纳入 PR Review 流程）
- [ ] 度量 SDD 效果指标：需求返工率、实现与 Spec 对齐度、首次通过率

**避免事项**：

- [ ] ❌ 不要让 AI Agent 自己写 Spec（环形验证风险）
- [ ] ❌ 不要跳过 Spec Review 直接进入编码
- [ ] ❌ 不要忽略模糊需求——"先让 AI 试试看"可能产生规模化错误

---

### P.2 测试与 QA：AI 时代的质量保障检查清单

**测试策略**：

- [ ] 区分"AI 生成的代码"与"人写的代码"的测试策略（前者需要更密集的边界测试）
- [ ] 引入异质模型审查（用不同于代码生成模型的 AI 做审查）
- [ ] 覆盖率指标补充：在行覆盖率之上增加分支覆盖率和突变覆盖率
- [ ] 建立"测试有效性"审计（定期检查哪些 Bug 在测试覆盖的代码中出现）

**环形验证防范**：

- [ ] 永远不要用同一模型生成代码和对应的测试
- [ ] 测试生成后由人类 Q A 或不同模型审查测试质量
- [ ] 定期运行突变测试（Mutation Testing）验证测试套件的真实有效性
- [ ] 对安全关键代码强制执行形式化验证或属性测试（Property-Based Testing）

**测试数据管理**：

- [ ] 定期更新测试数据集以覆盖 AI 生成代码的新模式
- [ ] 建立"回归测试套件"与"探索性测试套件"的分层体系
- [ ] AI 生成的测试用例需标记来源，便于追溯

**避免事项**：

- [ ] ❌ 不要仅凭覆盖率数字判断测试质量
- [ ] ❌ 不要让 AI 同时生成实现 + 单元测试且无人审查就合并
- [ ] ❌ 不要忽略 E2E 测试——单元测试的 AI 生成率高掩盖了集成层面的问题

---

### P.3 CI/CD：应对 PR 暴增的检查清单

**Code Review 流程调整**：

- [ ] 区分"AI 生成的 PR"和"人写的 PR"的 Review 策略
- [ ] AI 生成的 PR 强制通过 AI Code Review 工具（如 CodeRabbit）初审
- [ ] 定义"人类必须审查"的红线（如安全相关、核心业务逻辑、数据库 Schema 变更）
- [ ] 设置 PR 粒度上限（单 PR 超过 N 行/文件需要拆分）

**流水线健康度**：

- [ ] 监控 AI 生成代码的 CI 失败率（与人类代码对比）
- [ ] 建立"PR 审查队列深度"指标——如果持续增长，需要调整策略
- [ ] 定期审计 AI 生成的 PR 中被 Revert 的比例
- [ ] 实现"AI 自我审查"步骤——在提交 PR 前，Agent 自查常见问题

**安全门控**：

- [ ] 所有 AI 生成的 CI/CD 流程变更需人工审批
- [ ] 禁止 Agent 自主修改流水线配置（`.github/workflows/`、`Jenkinsfile` 等）
- [ ] 密钥和凭证不得出现在 Agent 可访问的上下文中

**避免事项**：

- [ ] ❌ 不要让 AI Code Review 完全替代人类审查
- [ ] ❌ 不要接受"看起来可以"的 AI 生成 PR——验证它的测试是否真的通过
- [ ] ❌ 不要忽略 CI 失败率上升的趋势信号

---

### P.4 安全工程：AI 时代安全防护检查清单

**防护体系**：

- [ ] 实施分层防御：输入过滤 → Agent 权限最小化 → 输出验证 → 人类审批
- [ ] Agent 的执行环境使用沙盒隔离
- [ ] 定期对 Agent 进行"对抗性测试"（尝试 Prompt Injection）
- [ ] 所有 AI 生成的代码变更在合并前需通过安全扫描

**Prompt Injection 防护**：

- [ ] 用户输入与系统指令严格分离（使用不同 Channel 或标签）
- [ ] 对来自外部源（Issue、PR 评论、用户输入）的内容进行清洗后再注入 Agent 上下文
- [ ] 关键指令使用不可覆盖的格式（如 System Prompt vs User Prompt 的层级区分）
- [ ] 限制 Agent 可访问的工具和 API 范围（最小权限原则）

**供应链安全**：

- [ ] AI 推荐的依赖包在安装前人工审查（防范 Slopsquatting）
- [ ] 监控 AI 生成代码中引入的外部依赖
- [ ] 依赖版本锁定，不允许 Agent 自主升级依赖

**避免事项**：

- [ ] ❌ 不要让 Agent 访问生产环境密钥
- [ ] ❌ 不要信任 Agent 的"安全自查"——始终独立验证
- [ ] ❌ 不要忽略 Agent 行为日志——定期审计异常行为模式

### P.5 可观测性：Agent 系统监控检查清单

**三支柱覆盖**：

- [ ] **Traces**：每次 Agent 调用的完整链路追踪（工具调用、API 耗时、Token 消耗）
- [ ] **Logs**：Agent 的思考过程（reasoning）、决策依据、错误和重试记录
- [ ] **Metrics**：成功率、延迟分布、Token 成本、幻觉率（如适用）

**关键告警**：

- [ ] Agent 连续失败率超过阈值（建议：>10%）
- [ ] Token 消耗异常飙升
- [ ] Agent 执行了其权限范围外的操作尝试
- [ ] 代码生成量与人类 Review 量的比值失衡

**成本归因**：

- [ ] 按 Feature/Team/Agent 维度的 Token 成本追踪
- [ ] 建立"AI 生成代码的 TCO"模型（Token 成本 + Review 成本 + 修复成本）
- [ ] 定期审计 ROI：AI 生成代码的维护成本 vs 传统开发成本

**避免事项**：

- [ ] ❌ 不要仅监控 Agent 是否"报错"——还要监控它在做什么
- [ ] ❌ 不要忽略"静默失败"——Agent 生成了代码但逻辑错误
- [ ] ❌ 不要让 Agent 的可观测性系统本身被 Agent 管理（监控与被监控需分离）

### P.6 综合转型路线图

| 阶段 | 时间框架 | 关键行动 | 成功指标 |
|------|----------|----------|----------|
| **试点** | 1-2 月 | 在 1-2 个非关键项目中引入 AI Agent + SDD | Spec→Code 对齐度 >80% |
| **推广** | 3-6 月 | 扩展到 5-10 个项目，建立指令文件规范和 Review 标准 | PR 审查时间不超过基准的 150% |
| **规范化** | 6-12 月 | 建立 AI 辅助开发的工程标准、合规流程、度量体系 | 安全事件数不因 AI 使用而增加 |
| **优化** | 持续 | 基于度量数据调整策略，引入新工具，更新指令文件 | AI 生成代码的返工率 <20% |

---

> **使用建议**：以上检查清单可根据团队规模、系统重要性和合规要求调整。建议从 P.1（SDD）和 P.4（安全）开始，因为它们是其他实践的基础。
>
> **更新频率**：每季度根据行业实践变化和技术演进攻订。
